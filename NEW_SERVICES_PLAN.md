# Condition Pages Plan — drkoren.skin

## Why This Matters

The SEO_PLAN.md identifies dedicated condition pages as the **single biggest lever** for on-site SEO improvement. Currently, high-volume conditions (psoriasis, eczema, rosacea, etc.) are buried as bullet points inside broad service pages. When someone searches "פסוריאזיס טיפול חיפה", Google matches the competitor's dedicated `/פסוריאזיס-psoriasis/` page over a general dermatology page where psoriasis is one bullet among many.

This is not about content quality — drkoren.skin's content is already stronger. It's about **keyword-to-page matching precision**.

### Israeli Market Context

- **Psoriasis**: ~300,000 patients in Israel (~3% of population). Israel is a global destination for Dead Sea psoriasis treatment, driving very high search volume. Dr. Koren leads the psoriasis clinic at Emek Medical Center — a major authority signal.
- **High sun exposure**: Israel's UV index is among the highest in the developed world, making skin cancer screening, actinic keratosis, and melasma particularly relevant.
- **Military service**: Universal conscription means young adults with acne, eczema, and stress-related conditions are a large patient population.
- **Immigration**: Diverse skin types (Ashkenazi, Mizrahi, Ethiopian) create varied dermatological needs.
- **Botox**: Strong demand in the Israeli aesthetic market. "בוטוקס חיפה" is a high-intent commercial query.
- **Herpes**: Very common in Israel (HSV-1 seroprevalence >90% in adults). Google Trends shows "הרפס" dominates Haifa dermatology searches at 67% share — the strongest local signal of any uncovered condition. Patients search for specialist treatment during outbreaks.
- **Vitiligo**: Sharply rising search trend nationally. Appears as a rising topic under "רופא עור" searches. Growing awareness and new treatment options (e.g., ruxolitinib cream) are likely driving increased search volume.

### Competitor Analysis

**drferaru.co.il** has 40+ individual condition pages, each targeting one specific keyword. Examples:
- `/פסוריאזיס-psoriasis/`
- `/אטופיק-דרמטיטיס-atopic-dermatitis/`
- `/רוזציאה-rosacea/`
- `/בוטוקס-botox/`

Many of these pages are thin (~500 words of generic medical info), but they still rank because they match the exact search query in the title, URL, H1, and meta description.

**Strategy**: We create fewer but substantially better pages. 13 excellent condition pages with genuine expert content, proper structured data, and strong internal linking will outperform 40 thin pages — especially under Google's Helpful Content Update.

### How Hebrew Searches Work

Israeli users search in three patterns:
1. **Transliterated Hebrew**: "פסוריאזיס" (Hebrew letters spelling the English medical term)
2. **Native Hebrew name**: "ספחת" (the actual Hebrew word for psoriasis)
3. **Mixed**: "פסוריאזיס טיפול" (transliterated condition + Hebrew "treatment")

Each condition page must include both the transliterated and native Hebrew names where applicable, plus the English term for bilingual searchers.

## Section 3: Implementation Steps

### Step 1: Create the `ConditionItem` Type

**File**: `types/condition.ts`

```typescript
export type ConditionItem = {
  slug: string                       // English-only URL slug (e.g., "psoriasis")
  title: string                      // Hebrew H1 (e.g., "פסוריאזיס (ספחת) — אבחון וטיפול")
  hebrewName: string                 // Primary Hebrew name (e.g., "פסוריאזיס")
  hebrewAlternate?: string           // Alternative Hebrew name (e.g., "ספחת")
  englishName: string                // English name for bilingual SEO (e.g., "Psoriasis")
  parentServiceSlug: string          // Links to parent service (e.g., "dermatology")
  metaTitle: string                  // SEO title, under 60 chars
  metaDescription: string            // SEO description, 140-160 chars, includes "חיפה"
  shortDescription: string           // 1-2 sentence summary
  content: {                         // Main page content sections
    overview: string                 // What is this condition (100-150 words)
    symptoms: string[]               // Key symptoms list
    diagnosis: string                // How Dr. Koren diagnoses (100-150 words)
    treatment: string                // Treatment approach (150-200 words)
    whenToSeeDoctor: string          // When to seek help (50-100 words)
  }
  faq: {
    question: string
    answer: string
  }[]                                // 3-5 condition-specific FAQs
  relatedConditions?: string[]       // slugs of related condition pages
  relatedServices?: string[]         // slugs of related service pages (beyond parent)
}
```

**Also update** `types/index.ts` to export the new type:
```typescript
export type { ConditionItem } from './condition'
```

### Step 2: Create the Conditions Data File

**File**: `lib/data/conditions.ts`

Create the data file with all condition objects. Start with Priority 1 (7 conditions), then add Priority 2 (6 conditions) in a second pass.

Content should be:
- Extracted and **expanded** from parent service pages (not copy-pasted)
- Written from Dr. Koren's perspective as a specialist
- 400-600 words total per condition
- FAQs must be distinct from the parent service page FAQs (no duplicates)

**Key rule**: Parent service pages provide an overview; condition pages go deeper into that specific condition.

### Step 3: Create the `/conditions/[slug]` Route

**File**: `app/conditions/[slug]/page.tsx`

Follow the exact same patterns as `app/services/[slug]/page.tsx`:

```
- generateStaticParams() — pre-renders all condition slugs
- generateMetadata() — per-condition title, description, canonical, OG
- Breadcrumb: Home > שירותי המרפאה > [Parent Service] > [Condition]
- Condition-specific JSON-LD via generateConditionStructuredData()
- CTA buttons (Phone + WhatsApp) at bottom — reuse ServiceCTAButtons
- Link back to parent service page
- Links to related conditions
- FAQ section with Schema.org markup
```

**URL structure**: `/conditions/[slug]` with English-only slugs, consistent with existing `/services/[slug]` pattern.

### Step 4: Add Structured Data Generator

**File**: `lib/seo/structured-data.ts`

Add a new `generateConditionStructuredData(condition)` function:

```typescript
export const generateConditionStructuredData = (condition: ConditionItem) => ({
  "@context": "https://schema.org",
  "@type": "MedicalWebPage",
  name: condition.title,
  description: condition.metaDescription,
  url: `${SITE.baseUrl}/conditions/${condition.slug}`,
  datePublished: "2026-02-19",
  dateModified: "2026-02-19",
  breadcrumb: {
    "@type": "BreadcrumbList",
    itemListElement: [
      { position: 1, name: "דף הבית", item: SITE.baseUrl },
      { position: 2, name: "שירותי המרפאה", item: `${SITE.baseUrl}/services` },
      { position: 3, name: parentServiceTitle, item: `${SITE.baseUrl}/services/${condition.parentServiceSlug}` },
      { position: 4, name: condition.title, item: `${SITE.baseUrl}/conditions/${condition.slug}` }
    ]
  },
  about: {
    "@type": "MedicalCondition",
    name: condition.hebrewName,
    alternateName: [condition.hebrewAlternate, condition.englishName].filter(Boolean),
    possibleTreatment: { "@type": "MedicalTherapy", name: condition.title }
  },
  author: { "@type": "Physician", name: SITE.name, url: SITE.baseUrl },
  speakable: {
    "@type": "SpeakableSpecification",
    cssSelector: ["h1", "[itemprop='description']", ".faq-answer:first-of-type"]
  }
})
```

Also generate per-condition `FAQPage` structured data from the condition's FAQ items.

### Step 6: Add Hub-and-Spoke Links

This is the most important step for SEO value. Both directions must be linked.

#### 6a: Condition pages link UP to parent service

Each condition page already links to its `parentServiceSlug` via the breadcrumb and an explicit "back to parent" link. No extra work needed.

#### 6b: Service pages link DOWN to condition pages

**File**: `lib/data/services.ts`

Add an optional `conditionSlug` field to bullet items. This requires a small type change:


Then in the dermatology service page, change:
```

```
to:
```typescript

{ text: 'סבוראיק דרמטיטיס (סבוריאה)', conditionSlug: 'seborrheic-dermatitis' },
```

**File**: `app/services/[slug]/page.tsx`

Update the bullet rendering logic to check for `conditionSlug` and render as `<Link href={/conditions/${conditionSlug}}>` instead of plain text.

#### 6c: Cross-link between related conditions

Use the `relatedConditions` field in `ConditionItem` to render a "related conditions" section at the bottom of each condition page (above the CTA).

## Section 4: Content Guidelines

### What Each Condition Page Needs

1. **H1**: Condition name in Hebrew (transliterated + native if applicable) + "אבחון וטיפול"
2. **Meta title**: Under 60 chars. Format: `[Condition Hebrew] - ד״ר תמר קורן | רופאת עור בחיפה`
3. **Meta description**: 140-160 chars. Must include condition name, "חיפה", and a call to action or differentiator.
4. **Content**: 400-600 words total, structured as:
   - Overview: What is this condition (100-150 words)
   - Symptoms: Bulleted list of key symptoms
   - Diagnosis: How Dr. Koren diagnoses this specifically (100-150 words)
   - Treatment: What treatment options are available (150-200 words)
   - When to see a doctor: Brief guidance (50-100 words)
5. **FAQs**: 3-5 questions, distinct from parent service page FAQs
6. **CTA**: Phone + WhatsApp buttons (reuse `ServiceCTAButtons`)
7. **JSON-LD**: `MedicalCondition` + `MedicalWebPage` + `FAQPage` schemas
8. **Internal links**: Back to parent service, to related conditions, to related services

### Avoiding Duplicate Content with Parent Service Pages

This is critical. Google penalizes duplicate content and it hurts both pages.

**Rule**: Parent service pages provide breadth; condition pages provide depth.

| Content Type | Parent Service Page | Condition Page |
|-------------|-------------------|----------------|
| Condition mention | Brief bullet or paragraph | Full dedicated content |
| Treatment process | Full step-by-step workflow | Condition-specific treatment details |
| FAQs | General service FAQs | Condition-specific FAQs (no overlap!) |
| Related conditions | Listed as bullets | Cross-linked to dedicated pages |
| CTA | Book appointment | Book appointment for [specific condition] |

**Example for Psoriasis:**
- **Dermatology service page**: Lists "פסוריאזיס" as a bullet under "מצבים נפוצים", links to `/conditions/psoriasis`
- **Psoriasis condition page**: Deep dive into types of psoriasis, triggers, Dr. Koren's approach as psoriasis clinic head at Emek, biologic therapy candidacy, Dead Sea treatment context, psoriasis-specific FAQs

### URL Slug Format

- English-only slugs: `/conditions/psoriasis`, `/conditions/atopic-dermatitis`
- Consistent with existing pattern: `/services/acne`, `/services/hair-loss`
- Lowercase, hyphen-separated
- No Hebrew in URLs (simplifies routing, avoids encoding issues, matches existing convention)

### Content Tone

- Professional but accessible Hebrew
- First-person plural where appropriate ("במרפאתנו")
- No unverified medical claims
- Must reflect Dr. Koren's actual treatment approach
- Include Dr. Koren's unique qualifications where relevant (e.g., psoriasis clinic head, Technion graduate)

## Architecture Decision: `/conditions/` vs `/services/`

### Decision

Conditions live at `/conditions/[slug]`, services live at `/services/[slug]`. These are separate route hierarchies.

### Rationale

- **Industry standard**: Mayo Clinic, Cleveland Clinic, WebMD all separate conditions from services in their URL hierarchy. No major medical site puts condition info under `/services/`.
- **Schema.org alignment**: `MedicalCondition` and `MedicalTherapy` are distinct Schema.org types. Matching this at the URL level strengthens structured data signals.
- **Hub-and-spoke SEO**: `/services/dermatology` is the hub, linking down to `/conditions/psoriasis`, `/conditions/atopic-dermatitis`, etc. Each spoke strengthens the hub AND ranks independently. A flat `/services/*` structure loses this multiplicative effect.
- **Search intent separation**: `/services/dermatology` targets "dermatologist Haifa" queries. `/conditions/psoriasis` targets "psoriasis treatment Haifa" queries. Separate URLs let Google serve the precisely matched result.
- **Google's YMYL/E-E-A-T**: Medical content that clearly distinguishes educational (condition) from procedural (service) content aligns with Google's quality rater guidelines.
- **No SEO penalty** for having more route segments — URL depth is identical either way.

### Conditions are NOT services

Conditions are **educational content about medical conditions**, linked from service pages. Services describe what the practice offers (procedures, workflows, pricing context). Conditions describe what the patient has (symptoms, diagnosis, treatment options).

### Discoverability

Conditions are surfaced through:
1. `/services` index page — renders conditions in a dedicated section automatically from the `conditions` array
2. Parent service page bullets — linked via `{ text, conditionSlug }` objects in `lib/data/services.ts`
3. Sitemap — auto-included via `conditions.map()` in `app/sitemap.ts`
4. Practice-info API — auto-included for AI agent discoverability

### Adding a new condition

1. Add condition object to `lib/data/conditions.ts`
2. Link from parent service page: add `{ text, conditionSlug }` to the service's `bullets` in `lib/data/services.ts`
3. Auto-appears on `/services` index page, sitemap, and API — no manual update needed
